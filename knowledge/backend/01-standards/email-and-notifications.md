# 邮件与通知标准（商业级必读）

> 注册验证、密码重置、订单状态、告警——通知是商业 app 的关键链路。发不出去、进垃圾箱、重复发、内容写死，都会影响转化与信任。

## 1. 用专业服务，别裸 SMTP

- 事务邮件用专业服务（SendGrid/SES/Postmark/Resend/阿里云邮件），获得送达率、退信处理、统计。
- **不要**用个人/裸 SMTP 发商业邮件（易进垃圾箱、无送达保障）。
- 配置 **SPF / DKIM / DMARC**（域名认证）提升送达率、防伪造。

## 2. 异步发送 + 可靠性

- 发送走**后台队列**（见后台任务标准），不阻塞请求；失败重试 + DLQ。
- **幂等**：同一事件不重复发（用事件 id 去重）——别让一次注册发三封验证邮件。
- 关键通知（密码重置、支付）记录发送状态，失败告警/兜底（如改发短信）。

## 3. 模板与内容

- 通知内容用**模板 + 变量**，不要在代码里硬编码拼字符串。
- 模板可维护、可本地化（i18n）；HTML 邮件提供纯文本 fallback。
- 链接用带签名/过期的 token（验证/重置链接），走 HTTPS，不泄露敏感信息于 URL。
- 文案清晰、品牌一致、有退订入口（营销类必须，合规）。

## 4. 多渠道与偏好

- 抽象**通知层**：同一事件可走邮件/短信/站内信/push，按用户偏好与紧急度选渠道。
- 用户可配置通知偏好（哪些通知、哪些渠道）；尊重退订/免打扰。
- 区分**事务通知**（必达：验证/支付/安全）与**营销通知**（可退订、限频）。

## 5. 合规与防滥用

- 营销邮件遵守反垃圾法规（CAN-SPAM/GDPR/个保法）：明确退订、真实发件人、不买卖名单。
- 发送限频，防被滥用刷验证码/轰炸；验证码类加冷却。
- 收集最小必要、可退订、可删除。

## 6. 反模式（出现即不合格）

- 在请求线程里同步发邮件让用户等；发送失败无重试无记录。
- 同一事件重复发（不幂等）；裸 SMTP 发商业邮件、无 SPF/DKIM。
- 邮件内容硬编码拼字符串、不可维护、无纯文本 fallback。
- 验证/重置链接无过期/无签名/明文敏感信息。
- 营销邮件无退订；验证码无限频可被轰炸。

## 7. 最低交付 checklist

- [ ] 用专业邮件服务 + SPF/DKIM/DMARC；不裸 SMTP。
- [ ] 异步队列发送 + 重试 + DLQ + 幂等去重；关键通知记录状态/告警兜底。
- [ ] 模板+变量、可 i18n、HTML+纯文本、链接带签名过期走 HTTPS。
- [ ] 通知层抽象多渠道 + 用户偏好；区分事务/营销，营销可退订限频。
- [ ] 合规（退订/最小收集）+ 验证码冷却防轰炸。
