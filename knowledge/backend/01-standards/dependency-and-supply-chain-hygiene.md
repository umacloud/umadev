# 依赖与供应链卫生规范（产品团队商业级必读）

> 现代应用 80% 以上的代码不是你写的，是你的依赖、以及依赖的依赖。每引入一个包，你就把自己产品的安全与稳定性，托付给了一群你从没见过的维护者。
> 这份规范不是给安全团队的攻防手册（那部分见 `security/01-standards/supply-chain-security`、`security/supply-chain-security`），而是给**做产品的工程团队**的日常纪律：怎么选依赖、怎么锁、多久更新一次、漏洞来了多快必须处理、怎么证明"我们装的就是我们以为的东西"。把供应链卫生当成**持续的运营工作**，而不是上线前补一次扫描。

## 1. 引入一个依赖前的准入

加一个依赖是一个**长期负债**决定，不是 `add` 一下那么轻：

- **先问要不要**：能用标准库 / 已有依赖几十行解决的，不要为此引入一个大树。每个依赖都带来传递依赖、漏洞面、升级负担。
- **评估健康度**：维护是否活跃、有没有在修安全问题、发布是否规律、单一维护者还是有组织、下载与采用规模、传递依赖体量、许可证是否可用。一次性小工具尤其警惕。
- **警惕仿冒**：包名与流行包高度相似（拼写仿冒）、刚发布不久、下载量极低却出现在依赖里——可能是仿冒/混淆攻击，引入前核对来源。
- **锁定来源**：内部包用组织 scope/命名空间，配置内部仓库优先，防止公共仓库同名包被优先解析（依赖混淆）。

## 2. 锁定：装的就是你以为的

- **必须有锁文件**：每个项目都有锁文件并**提交版本库**，锁定**全部**依赖（含传递依赖）的精确版本。"只锁直接依赖"等于没锁——攻击与破坏性变更大多来自传递层。
- **锁文件带完整性校验**：锁文件应包含哈希/完整性摘要，安装时校验，保证下载到的产物字节级一致（防中间篡改与仓库投毒）。
- **CI 用锁文件做可复现安装**：CI/构建用"严格按锁文件安装"的命令（不允许隐式升级），保证构建可复现；锁文件变更必须随 PR 一起评审。
- **锁文件 diff 要被看见**：依赖升级的锁文件 diff 进入代码评审视野，新增/跳变的传递依赖能被注意到，而不是无人看的几千行变更。

## 3. 更新节奏：与其攒大版本，不如小步常更

不更新和乱更新都是错的。攒着不更，等到出 0-day 才被迫一次性跨多个大版本，风险与工作量都爆炸。

| 更新类型 | 建议节奏 | 处理方式 |
|---|---|---|
| 安全补丁 | 立即（按分诊 SLA） | 优先合入，单独发布 |
| 补丁/次版本 | 每周/每两周批量 | 自动化 PR + 测试通过即合 |
| 主版本（破坏性） | 计划性、单独处理 | 读迁移说明、单独 PR、充分回归 |
| 已弃用/无维护依赖 | 主动排期替换 | 列入技术债，定替换计划 |

- **自动化更新 PR**：用自动化机器人（依赖更新 bot）持续开升级 PR，配合 CI 测试，把"更新"变成低摩擦的日常，而不是季度性大扫除。
- **分组与限流**：把低风险更新分组合并，限制同时打开的 PR 数，避免淹没评审。
- **更新有测试护栏**：依赖升级必须跑全量测试 + 关键路径回归后才合，杜绝"升级即合"导致悄悄破功能。

## 4. SBOM 与可追溯（provenance）

- **每次发布生成 SBOM**：构建产出软件物料清单（标准格式），完整列出制品包含的所有组件与版本，存入制品仓库随版本归档。
- **SBOM 是持续资产不是一次性文件**：新漏洞披露时，用历史 SBOM 反查"我们哪些线上版本受影响"，而不是临时去翻每个服务的依赖。
- **出处可证明**：能证明制品由可信流水线、从可信源码、用锁定的依赖构建而来（构建出处/签名）。生产部署侧校验制品来源与完整性，拒绝来路不明的产物。
- **依赖与制品绑定**：每个发布版本能精确回答"用了哪些依赖的哪些版本"，出事可定位影响范围。

## 5. 漏洞分诊：有 SLA，不是看到就慌、看不到就拖

扫描器会报一堆，**关键是分诊**——按真实风险定优先级和处理时限，而不是被 CVSS 数字牵着走。

- **持续扫描**：CI 中集成依赖漏洞扫描；并**定时**对已发布版本重扫（今天没漏洞的依赖明天可能被披露）。
- **按可达性 + 严重度分诊**：同一个 CVE，在你产品里**是否真的可达/可利用**（在调用路径上吗？是 dev 依赖吗？输入可控吗？）决定真实风险。结合严重度与可达性定级，避免对不可达漏洞过度反应、对可达漏洞低估。
- **明确处理 SLA**：给每个等级定响应时限（例：可利用的严重漏洞按小时计、高危按天计、中低危排期）。SLA 写进规范并被度量。
- **处置方式**：升级修复版本优先；无修复版本时评估临时缓解（关闭受影响功能/加防护/降级）；确认不可达可记录**带理由和到期日的例外**，到期复审，绝不无限期忽略。
- **CI 阻断要分级**：对"可达的高危/严重"阻断构建；对噪音类降级为告警，避免一刀切让团队学会绕过门禁。

## 6. 许可证与合规卫生

- **许可证可见**：知道每个（含传递）依赖的许可证；CI 检查是否引入与产品分发模式冲突的许可证（如对闭源产品引入强 copyleft）。
- **禁用清单**：维护不允许的许可证清单，新增依赖触发即阻断或人工审批。
- **可向客户提供清单**：商业合规场景下能提供组件与许可证清单（SBOM 复用）。

## 7. 反模式（出现即不合格）

1. **没有锁文件 / 只锁直接依赖**：构建不可复现，传递依赖随时漂移、可被投毒。
2. **攒着不更新**：依赖长期不动，出 0-day 才被迫跨多个大版本紧急升级。
3. **升级即合不测试**：依赖 PR 不跑回归就合，悄悄破坏功能。
4. **扫描噪音 + 一刀切阻断**：不分诊、不看可达性，门禁噪音大到团队学会绕过。
5. **看到漏洞无 SLA**：处理全凭心情，可利用的严重漏洞拖几周。
6. **只上线前扫一次**：不对已发布版本持续重扫，新披露漏洞无人发现。
7. **dev 依赖无人管**：以为开发期依赖不影响产品，忽视其漏洞与投毒风险。
8. **来路不明的制品直接上线**：不校验出处与完整性，无 SBOM，出事查不出影响范围。

## 8. 最低交付 checklist

- [ ] 引入依赖前评估健康度/维护性/许可证/传递体量，警惕仿冒与混淆。
- [ ] 每个项目有锁文件并入库，锁定全部（含传递）依赖 + 完整性校验；CI 严格按锁文件可复现安装。
- [ ] 自动化更新 PR + 测试护栏，小步常更；主版本计划性单独处理；弃用依赖列技术债排期替换。
- [ ] 每次发布生成并归档 SBOM，可用历史 SBOM 反查受影响线上版本。
- [ ] 制品出处可证明，生产侧校验来源与完整性，依赖与发布版本绑定可追溯。
- [ ] CI 集成漏洞扫描 + 对已发布版本定时重扫；按可达性+严重度分诊，分级处理有 SLA。
- [ ] 漏洞例外须带理由与到期日并复审；可达的严重漏洞阻断发布。
- [ ] 许可证可见、有禁用清单、可按需输出组件与许可证清单。
