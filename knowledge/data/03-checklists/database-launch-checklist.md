# 数据库上线检查清单

## 概述

本清单用于数据库服务从开发/测试环境迁移到生产环境前的系统化审查。覆盖备份恢复、监控告警、连接池、索引、安全和性能基线六大维度。所有 MUST 级别项未通过则阻止上线。

适用数据库：PostgreSQL、MySQL、MongoDB。特定数据库的差异会标注说明。

---

## 1. 备份与恢复（Backup & Recovery）

### MUST

- [ ] 自动备份策略已配置并验证（至少每日全量备份）
- [ ] 备份文件存储在独立于数据库服务器的位置（异地/异机房）
- [ ] 增量备份 / WAL 归档已启用（PostgreSQL: archive_mode=on; MySQL: binlog）
- [ ] 备份恢复已实际测试，恢复时间满足 RTO 要求
- [ ] 恢复后数据完整性已验证（行数对比 + 关键数据校验）
- [ ] RPO（恢复点目标）已定义且备份频率满足要求
- [ ] 备份文件加密存储（AES-256 或等效）

### SHOULD

- [ ] 支持时间点恢复（PITR: Point-In-Time Recovery）
- [ ] 备份保留策略已定义（如日备份保留 7 天，周备份保留 4 周，月备份保留 12 月）
- [ ] 每月执行一次备份恢复演练
- [ ] 备份完成/失败有自动通知
- [ ] 备份大小趋势监控（异常增长告警）
- [ ] 跨区域备份副本（灾备要求）

## 2. 监控与告警（Monitoring & Alerting）

### MUST

- [ ] 数据库实例存活监控（每 30 秒检测，连续 3 次失败告警）
- [ ] CPU 使用率 > 80% 持续 5 分钟告警
- [ ] 内存使用率 > 85% 告警
- [ ] 磁盘使用率 > 80% 告警，> 90% 紧急告警
- [ ] 活跃连接数 > 连接池上限的 80% 告警
- [ ] 慢查询数量突增告警（比基线增加 200%）
- [ ] 主从复制延迟 > 10 秒告警

### SHOULD

- [ ] Grafana / Datadog 仪表板已创建，包含以下面板：
  - QPS（查询/秒）
  - 活跃连接数
  - 缓存命中率（Buffer Cache Hit Ratio）
  - 事务提交/回滚比
  - 表膨胀（PostgreSQL dead tuple 比例）
  - 锁等待数量和时长
- [ ] 告警分级：P1（数据库宕机/数据丢失）、P2（性能劣化）、P3（非紧急异常）
- [ ] 告警升级策略：P1 > 5 分钟未响应自动电话通知
- [ ] 长事务（> 5 分钟）监控告警
- [ ] 死锁检测告警
- [ ] 表大小增长趋势监控

## 3. 连接池（Connection Pool）

### MUST

- [ ] 应用层连接池已配置（不使用数据库直连）
- [ ] 最大连接数已计算并设置合理值
  ```
  公式：max_connections = (应用实例数 × 每实例连接池大小) + 预留管理连接(10)
  ```
- [ ] 连接池大小已根据压测结果调优
- [ ] 连接超时设置合理（获取连接超时 5s，空闲超时 300s）
- [ ] 连接健康检查已启用（获取连接前验证连接可用）
- [ ] 数据库端 max_connections 大于所有连接池之和

### SHOULD

- [ ] 使用连接池中间件（PgBouncer / ProxySQL）减少数据库端连接数
- [ ] 连接池使用率指标暴露到监控系统
- [ ] 连接泄漏检测已启用（长时间未归还的连接告警）
- [ ] 读写分离场景下读/写连接池独立配置
- [ ] 连接池预热：应用启动时预创建最小连接数
- [ ] PgBouncer 使用 transaction mode（而非 session mode）以提高连接复用

## 4. 索引与查询（Index & Query）

### MUST

- [ ] 所有外键列已建立索引
- [ ] 高频查询的 WHERE / JOIN / ORDER BY 列已建立索引
- [ ] 唯一性约束通过 UNIQUE INDEX 实现（而非应用层检查）
- [ ] 无重复索引（被更宽复合索引覆盖的索引已删除）
- [ ] 生产环境无 `SELECT *` 的查询（指定列名）
- [ ] 分页查询使用游标分页（keyset pagination）而非大 OFFSET

### SHOULD

- [ ] 未使用的索引已清理（`pg_stat_user_indexes.idx_scan = 0`）
- [ ] 复合索引遵循最左匹配原则
- [ ] 查询执行计划中无全表扫描（数据量 > 10000 行的表）
- [ ] 批量操作使用 COPY / LOAD DATA 而非逐行 INSERT
- [ ] ORM 生成的 SQL 已审查（无 N+1 查询）
- [ ] 定期 ANALYZE 更新统计信息（PostgreSQL: autovacuum 已启用）

## 5. 安全（Security）

### MUST

- [ ] 数据库不使用默认端口（PostgreSQL 5432 / MySQL 3306）或有防火墙保护
- [ ] 数据库不对公网开放，仅允许应用服务器 IP 访问
- [ ] 应用使用独立的数据库账户（非 root/superuser）
- [ ] 应用账户权限最小化（仅 SELECT/INSERT/UPDATE/DELETE，无 DROP/CREATE）
- [ ] 数据库密码复杂度满足要求（≥ 16 字符，含大小写+数字+特殊字符）
- [ ] 数据库密码通过环境变量或 Secret Manager 注入（不硬编码在代码中）
- [ ] 传输加密已启用（SSL/TLS 连接）
- [ ] 敏感数据列已加密存储（密码哈希、身份证号、银行卡号）

### SHOULD

- [ ] 数据库审计日志已启用（DDL 操作 + 敏感表访问）
- [ ] 定期轮换数据库密码（≤ 90 天）
- [ ] 静态数据加密（Encryption at Rest）已启用
- [ ] 数据库用户定期审查（移除不再需要的账户）
- [ ] SQL 注入防护验证：所有应用查询使用参数化查询
- [ ] 行级安全策略（RLS）用于多租户隔离（PostgreSQL）
- [ ] 备份文件访问权限限制（仅 DBA 可访问）

## 6. 性能基线（Performance Baseline）

### MUST

- [ ] 关键查询的 P50/P95/P99 延迟已记录
- [ ] 压力测试已完成，确认最大 QPS 满足业务预期的 2 倍
- [ ] 压力测试期间无死锁、连接耗尽、OOM 发生
- [ ] 事务隔离级别已确认（默认 Read Committed，需要更高级别的场景已标注）
- [ ] 大表（> 100 万行）的查询性能已单独验证
- [ ] vacuum / autovacuum 参数已调优（PostgreSQL）

### SHOULD

- [ ] 基线数据存档以供后续对比
- [ ] 数据库配置参数已根据硬件资源调优：
  - `shared_buffers`：物理内存的 25%（PostgreSQL）
  - `effective_cache_size`：物理内存的 75%（PostgreSQL）
  - `work_mem`：根据并发连接数计算
  - `innodb_buffer_pool_size`：物理内存的 70%（MySQL）
- [ ] 连接风暴测试：模拟所有应用实例同时重启
- [ ] 故障切换测试：主库宕机后副本自动提升验证
- [ ] 数据增长预估：按当前增速计算 6 个月后的存储和性能影响

## Agent Checklist

以下为 AI Agent 在数据库上线审查时必须逐项验证的硬约束：

- [ ] 确认备份策略已配置：检查 cron job / 云服务备份配置
- [ ] 确认可以从最近一次备份恢复到测试环境
- [ ] 确认监控系统可收到数据库指标（连接数、QPS、延迟）
- [ ] 确认连接池配置存在且参数合理
- [ ] 确认所有外键列有对应索引
- [ ] 确认应用数据库账户非 superuser 且权限最小化
- [ ] 确认数据库密码不在代码仓库中明文出现
- [ ] 确认 SSL 连接已启用
- [ ] 确认关键查询性能基线已记录
- [ ] 生成上线检查报告并附在发布 ticket 中
