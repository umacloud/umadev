# PR 检查清单 (Pull Request Checklist)

> 适用场景：代码评审、合并请求审核、自动化 CI 门禁配合使用。
> 通过标准：所有 MUST 项通过方可合并；SHOULD 项未通过需在 PR 描述中说明理由。

---

## 1. 变更范围与描述

- [ ] **PR 标题清晰且符合规范** — 使用 Conventional Commits 格式（feat / fix / refactor / docs / test / chore），标题 <= 72 字符。
- [ ] **变更目的单一且明确** — 一个 PR 只解决一个问题或实现一个功能，不混合无关变更。
- [ ] **PR 描述完整** — 包含变更背景、实现方案概述、影响范围、测试方式，关联 Issue / Ticket 编号。
- [ ] **Breaking Change 已标注** — 如有不兼容变更，在 PR 描述和 commit message 中明确标注 `BREAKING CHANGE`。
- [ ] **变更规模合理** — 新增 / 修改代码 <= 400 行（不含自动生成代码），超大 PR 应拆分。
- [ ] **截图 / 录屏已附加**（如涉及 UI）— 变更前后对比截图已附加，覆盖关键状态。

---

## 2. 代码质量

### 2.1 可读性

- [ ] **命名清晰有意义** — 变量 / 函数 / 类 / 模块命名准确表达意图，无 `tmp` / `data` / `handler2` 等模糊名称。
- [ ] **函数职责单一** — 每个函数只做一件事，长度 <= 50 行（不含空行和注释），超长函数已拆分。
- [ ] **无重复代码** — 不引入 copy-paste 代码块，公共逻辑已提取为共享函数或模块。
- [ ] **注释说明 Why 而非 What** — 注释解释业务背景和设计决策，不重复描述代码逻辑。
- [ ] **魔术数字已消除** — 数值字面量提取为命名常量，含义一目了然。

### 2.2 设计

- [ ] **符合项目架构规范** — 新代码放在正确的模块 / 层级，遵循已有的分层与依赖方向。
- [ ] **接口设计合理** — 公开接口参数最小化，返回值类型明确，错误处理一致。
- [ ] **无过度设计** — 不引入当前不需要的抽象层、不提前优化，遵循 YAGNI 原则。
- [ ] **向后兼容** — API / 数据格式变更向后兼容，或有明确的迁移路径和版本计划。

### 2.3 代码规范

- [ ] **Lint 检查通过** — Ruff / ESLint / Prettier 等格式化与规范检查零错误。
- [ ] **类型检查通过** — MyPy / TypeScript 类型检查无新增错误。
- [ ] **无 TODO/FIXME/HACK 残留** — 临时代码已清理，或关联了对应的 Issue 进行后续跟踪。
- [ ] **import 排序正确** — 导入按标准库 / 三方库 / 项目内部排序，无未使用的导入。

---

## 3. 测试

### 3.1 测试覆盖

- [ ] **新增代码有对应测试** — 业务逻辑新增 / 修改的代码有单元测试覆盖，覆盖率不低于既有基线。
- [ ] **边界条件已覆盖** — 空值、零值、最大值、异常输入、并发场景的测试用例已编写。
- [ ] **错误路径已测试** — 异常抛出、错误码返回、降级逻辑均有测试用例验证。
- [ ] **回归测试已通过** — 既有测试套件全部通过，无因本次变更导致的测试失败。

### 3.2 测试质量

- [ ] **测试独立可重复** — 测试不依赖执行顺序或外部状态，可单独运行且结果稳定。
- [ ] **断言具体明确** — 不使用 `assertTrue(result)` 式模糊断言，校验具体字段和值。
- [ ] **Mock 范围合理** — 仅 Mock 外部依赖（HTTP / DB / 第三方），不 Mock 被测试对象的内部方法。
- [ ] **测试命名描述行为** — 测试函数名描述被测行为和预期结果，如 `test_login_fails_with_invalid_password`。

### 3.3 集成与端到端

- [ ] **集成测试已补充**（如涉及跨模块交互）— 模块间交互的关键路径有集成测试覆盖。
- [ ] **E2E 测试已更新**（如涉及用户流程变更）— 核心用户流程的 E2E 测试已同步更新。

---

## 4. 文档

- [ ] **代码注释已更新** — 修改的函数 / 类的 docstring 与实际行为一致。
- [ ] **API 文档已同步** — 接口变更同步更新 OpenAPI / Swagger / GraphQL Schema 文档。
- [ ] **README 已更新**（如涉及使用方式变更）— 安装步骤、配置项、使用示例已同步更新。
- [ ] **CHANGELOG 已更新** — 用户可感知的变更已记录在 CHANGELOG 中。
- [ ] **架构决策已记录**（如涉及设计变更）— ADR（Architecture Decision Record）已创建或更新。

---

## 5. 安全

- [ ] **无敏感信息提交** — 代码中无硬编码的密钥、Token、密码、内部 URL。
- [ ] **用户输入已校验** — 新增的外部输入已做白名单校验或参数化处理。
- [ ] **权限检查已覆盖** — 新增的接口 / 资源访问有对应的认证和授权校验。
- [ ] **SQL 查询参数化** — 新增 / 修改的数据库查询使用参数化查询，无字符串拼接。
- [ ] **依赖安全** — 新增依赖无已知 Critical / High 漏洞（npm audit / pip-audit 通过）。
- [ ] **日志不含敏感数据** — 新增日志语句不输出用户密码、Token、PII 等敏感信息。

---

## 6. 性能

- [ ] **无 N+1 查询** — 数据库访问无循环内单条查询的模式，使用批量查询或预加载。
- [ ] **无不必要的全表扫描** — 新增查询已 EXPLAIN 分析，必要字段已建索引。
- [ ] **大数据量处理已分页/流式** — 返回列表数据有分页，大文件处理使用流式读取。
- [ ] **缓存策略合理** — 频繁访问的不变数据已缓存，缓存键与失效策略已定义。
- [ ] **无内存泄漏风险** — 事件监听器已正确清理，大对象引用在使用后释放。
- [ ] **异步操作已正确处理** — Promise / async-await 无未捕获的异常，无 fire-and-forget 的关键操作。

---

## 7. 兼容性

- [ ] **API 版本兼容** — 接口变更不影响现有客户端，或有版本号隔离。
- [ ] **数据库 Schema 兼容** — DDL 变更可前向兼容（先加后删，不改类型），migration 可回滚。
- [ ] **配置变更兼容** — 新增配置有默认值，删除配置有迁移路径。
- [ ] **浏览器 / 运行时兼容**（如前端变更）— 在目标浏览器 / Node.js 版本上测试通过。
- [ ] **国际化兼容**（如涉及文案）— 新增文案已抽取为 i18n 资源，不硬编码中文 / 英文。

---

## 8. 运维影响

- [ ] **回滚方案已确认** — 本次变更可通过 revert commit 或功能开关快速回滚。
- [ ] **Feature Flag 已配置**（如涉及灰度）— 新功能可通过 Feature Flag 控制开关，默认关闭。
- [ ] **监控指标已添加** — 新增功能的关键路径有延迟 / 错误率 / 调用量指标。
- [ ] **告警规则已配置** — 新增指标的异常阈值已配置对应告警。
- [ ] **日志可追踪** — 关键业务操作有结构化日志，包含 traceId 和业务上下文。

---

## PR 合并前最终确认

- [ ] CI 流水线全部通过（lint + type check + test + security scan）。
- [ ] 至少 1 位 reviewer 已 Approve。
- [ ] 所有 review comment 已 resolve。
- [ ] 分支已 rebase 到最新 main，无冲突。
- [ ] PR 描述中的 checklist 已全部勾选或注明豁免理由。

## Agent Checklist

供 AI Agent 在创建或审核 PR 时使用的自检项：

- [ ] 已检查 PR 标题符合 Conventional Commits 格式。
- [ ] 已核对变更文件列表，确认无意外文件被包含（如 .env / lock 文件 / 构建产物）。
- [ ] 已逐文件 review 代码差异，非仅检查 CI 是否通过。
- [ ] 已验证新增测试实际覆盖了变更代码的核心路径。
- [ ] 已检查是否有未处理的 review comment。
- [ ] 已确认变更不引入安全风险（密钥泄露 / 注入 / 越权）。
- [ ] 审核结果已记录，包含通过项和待修复项清单。
