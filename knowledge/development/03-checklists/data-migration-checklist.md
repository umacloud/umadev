# 数据迁移检查清单 (Data Migration Checklist)

> 适用场景：数据库 Schema 变更、数据库引擎迁移、数据中心迁移、数据格式转换、大规模数据回填。
> 通过标准：所有 MUST 项全部通过方可执行生产迁移；SHOULD 项未通过需迁移负责人签字确认。

---

## 1. 迁移规划

### 1.1 范围与目标

- [ ] **迁移范围已明确** — 涉及的数据库 / 表 / 字段 / 记录范围已列出，有明确的包含和排除列表。
- [ ] **迁移目标已量化** — 预期迁移的数据量（行数 / 存储大小）、迁移完成时间已估算。
- [ ] **迁移类型已确定** — 在线迁移（零停机）/ 停机迁移 / 双写渐进迁移，方案已选定并论证。
- [ ] **迁移依赖已梳理** — 上下游服务、定时任务、数据消费方的影响已评估。
- [ ] **成功标准已定义** — 迁移完成的判定条件已量化（如数据一致率 >= 99.99%，性能不退化）。

### 1.2 时间窗口

- [ ] **迁移时间窗口已确定** — 选择业务低峰期，预留足够的执行 + 验证 + 回滚时间。
- [ ] **时间里程碑已拆分** — 准备阶段、执行阶段、验证阶段、切换阶段的时间节点已标记。
- [ ] **超时策略已定义** — 迁移超过预期时间 150% 时的决策流程（继续 / 暂停 / 回滚）已确定。

---

## 2. 迁移前准备

### 2.1 数据分析

- [ ] **源数据质量已评估** — 空值、异常值、重复数据、编码问题已统计并处理。
- [ ] **数据分布已分析** — 大表行数、索引大小、热点分布已确认，用于估算迁移耗时。
- [ ] **Schema 差异已映射** — 源和目标的 Schema 差异已列出，字段类型转换规则已定义。
- [ ] **外键与约束已梳理** — 外键依赖关系已理清，迁移顺序按依赖拓扑排序。
- [ ] **触发器 / 存储过程已评估** — 迁移过程中触发器是否需要临时禁用已确认。

### 2.2 环境准备

- [ ] **目标环境已就绪** — 目标数据库实例已创建，配置（字符集 / 排序规则 / 参数组）已对齐。
- [ ] **网络连通性已验证** — 源和目标数据库之间的网络延迟和带宽满足迁移需求。
- [ ] **存储空间已确认** — 目标存储空间 >= 源数据大小的 2x（预留临时表和日志空间）。
- [ ] **权限已配置** — 迁移账号具有源端 SELECT 和目标端 INSERT / UPDATE / DDL 权限。

### 2.3 脚本与工具

- [ ] **迁移脚本已编写** — 迁移脚本（SQL / ETL / 自定义程序）已完成，代码已 review。
- [ ] **脚本幂等性已保证** — 迁移脚本可安全重复执行，不产生重复数据或错误。
- [ ] **分批策略已实施** — 大表迁移按主键 / 时间范围分批执行，单批 <= 10000 行，避免长事务。
- [ ] **进度记录机制已实施** — 迁移进度可实时查看（已迁移行数 / 百分比 / 预计剩余时间）。
- [ ] **断点续传已支持** — 迁移中断后可从断点恢复，不需要从头开始。

---

## 3. 数据校验

### 3.1 迁移前校验

- [ ] **源数据快照已保存** — 迁移前的关键统计数据已记录（总行数 / 校验和 / 关键字段汇总值）。
- [ ] **抽样数据已导出** — 随机抽取 >= 1000 条记录用于迁移后逐行比对。
- [ ] **业务规则校验已定义** — 关键业务规则（如余额非负、状态机合法、关联关系完整）的校验查询已编写。

### 3.2 迁移后校验

- [ ] **行数一致性校验** — 源和目标的各表行数一致（允许偏差范围已定义）。
- [ ] **数据完整性校验** — 主键唯一性、外键完整性、非空约束均通过。
- [ ] **字段值校验** — 抽样数据逐字段比对通过，关键汇总值（SUM / COUNT / MAX / MIN）一致。
- [ ] **业务规则校验** — 迁移后数据满足所有预定义的业务规则。
- [ ] **索引与约束校验** — 目标表的索引、唯一约束、检查约束已正确创建。
- [ ] **编码与精度校验** — 中文 / 特殊字符未乱码，数值精度未丢失。

### 3.3 性能校验

- [ ] **查询性能对比** — 核心查询在目标库上的执行计划和耗时与源库一致或更优。
- [ ] **索引命中率正常** — 关键查询使用了预期的索引，无全表扫描。
- [ ] **应用性能验证** — 应用连接目标库后，核心接口延迟在可接受范围内。

---

## 4. 双写与切换（在线迁移专用）

- [ ] **双写逻辑已实现** — 写操作同时写入源和目标，或通过 CDC（Change Data Capture）实时同步增量。
- [ ] **双写一致性已验证** — 双写期间源和目标的数据差异 <= 可接受延迟（如 < 5s）。
- [ ] **读切换策略已定义** — 灰度切换读流量（如先切查询 -> 再切写入），每步有验证点。
- [ ] **写切换策略已定义** — 写流量切换窗口最小化，切换时的在途请求处理方案已确认。
- [ ] **切换回退机制** — 切换后发现问题可快速回退到源库，回退操作 <= 5 分钟。

---

## 5. 回滚方案

- [ ] **回滚脚本已编写** — 数据回滚脚本已完成，可将目标数据恢复到迁移前状态。
- [ ] **回滚触发条件已定义** — 明确什么情况下触发回滚（如数据不一致率 > 0.01%，应用错误率 > 5%）。
- [ ] **回滚时间已评估** — 回滚操作预计耗时已估算，在可接受范围内。
- [ ] **回滚已演练** — 在预发布环境执行过完整的迁移 + 回滚流程。
- [ ] **回滚后业务影响已评估** — 回滚后的数据损失范围（迁移后新增的数据如何处理）已有方案。
- [ ] **备份已创建** — 迁移前的数据库全量备份已完成，备份可用性已验证。

---

## 6. 对业务的影响评估

- [ ] **业务方已知晓** — 迁移时间 / 影响范围 / 预期停机（如有）已通知业务方。
- [ ] **定时任务已暂停** — 迁移期间相关的定时任务 / Cron Job 已暂停，迁移后恢复。
- [ ] **消息消费已暂停**（如需要）— 迁移期间相关消息队列消费者已暂停，积压量在可控范围。
- [ ] **缓存失效策略已定义** — 迁移后缓存中的旧数据清理方案已确定（全量清除 / 逐步过期）。
- [ ] **下游通知已发送** — 数据消费方（报表 / 分析 / 同步任务）已知悉迁移计划。

---

## 7. 人员与沟通

- [ ] **迁移负责人已指定** — 有明确的迁移协调人，对迁移过程全程负责。
- [ ] **DBA 支持已确认** — DBA 在迁移窗口内可提供支持。
- [ ] **值班安排已确认** — 迁移后 >= 4 小时有工程师值守观察。
- [ ] **升级路径已明确** — 迁移异常时的升级联系人和决策链已确认。
- [ ] **沟通渠道已建立** — 迁移过程的实时沟通群组 / 频道已创建。

---

## 8. 迁移后收尾

- [ ] **源库处理方案已确定** — 迁移完成后源库保留时间已定义（建议 >= 7 天只读保留）。
- [ ] **监控已切换** — 监控和告警已指向目标库，源库监控降级为备用。
- [ ] **连接串已全部切换** — 所有应用 / 脚本 / 定时任务的数据库连接已指向目标。
- [ ] **迁移文档已归档** — 迁移方案、执行日志、校验结果、回滚记录已存档。
- [ ] **事后复盘已排期** — 迁移完成后一周内安排复盘，总结经验和改进点。

## Agent Checklist

供 AI Agent 在执行或监督数据迁移时使用的自检项：

- [ ] 已确认迁移脚本在预发布环境完整执行通过。
- [ ] 已核对源和目标的 Schema 差异映射无遗漏。
- [ ] 已验证数据校验覆盖行数 / 完整性 / 业务规则三个维度。
- [ ] 已确认回滚方案可实际执行（非纸面方案）。
- [ ] 已验证迁移后应用性能无退化。
- [ ] 已确认所有定时任务和消息消费者已正确暂停 / 恢复。
- [ ] 迁移执行日志和校验报告已存档至 `output/` 目录。
