# 测试策略与分层规范（商业级必读）

> 商业级项目必须有自动化测试，且**按架构分层去测**——不同层测不同东西，比例呈金字塔。不写测试、或全堆在 e2e、或测实现细节，都是不合格。

## 1. 测试金字塔（比例）

```
        /\        E2E（少量，关键业务流）         慢、脆、贵
       /  \       Integration（适量，服务+仓储+真DB）
      /____\      Unit（大量，领域逻辑/纯函数）    快、稳、便宜
```

- **大量单元测试**（毫秒级、无 IO）+ **适量集成测试** + **少量端到端**。倒金字塔（一堆 e2e、几乎没单测）是反模式：慢、脆、定位难。

## 2. 各层测什么（对应分层架构）

| 层 | 测什么 | 类型 |
|---|---|---|
| 领域层(entity/VO/纯函数) | 业务规则、不变量、状态机、计算 | 单元（无 mock，纯内存）|
| 应用层(service/use-case) | 用例编排、事务、错误路径；mock 掉 repository/gateway 接口 | 单元/集成 |
| 基础设施层(repository) | 真实读写、映射、查询正确性 | 集成（真实 DB，用 testcontainers/内存库）|
| 接口层(controller/API) | 路由、校验、状态码、错误信封、鉴权 | 集成/契约 |
| 关键业务流 | 注册→下单→支付 等端到端 happy + 关键失败 | E2E |

## 3. 写法规范

- **AAA / Given-When-Then**：准备(Arrange)→执行(Act)→断言(Assert)，结构清晰。
- 一个测试只验**一个行为**；测试名说清"在什么条件下、做什么、期望什么"。
- 测**行为/契约**，不测私有实现细节（否则重构就红）。
- 覆盖**正常 + 边界 + 错误**路径，不只 happy path。
- 测试相互独立、可任意顺序、可并行；不依赖执行顺序或共享可变状态。
- 用测试替身（test double）替换外部依赖：mock 接口、stub 返回、fake 内存实现；不要在单测里打真网络/真第三方。

## 4. 数据与环境

- 集成测试用一次性真实依赖（testcontainers / 内存 Postgres / sqlite），每个用例自带数据、跑完清理，不污染。
- 不依赖生产/共享测试库；不依赖"先跑某个测试"。
- 时间、随机、UUID 等不确定源要可注入/可固定，避免 flaky。

## 5. 覆盖率与质量门

- 关注**关键路径覆盖**（业务核心、钱/权限相关）而非盲目追 100% 行覆盖。
- 合理目标：领域/服务层高覆盖（如 ≥80%），整体设阈值并在 CI 卡住。
- 覆盖率是下限不是目标；高覆盖 + 烂断言 = 假安全。

## 6. CI 集成

- 每次 PR/push 自动跑 lint + 单元 + 集成；e2e 在合并前或 nightly。
- 测试失败必须**阻断合并**；flaky 测试要修或隔离，不允许"重跑就过"长期存在。
- 测试要快（单测整套秒级），慢测试拆分/并行。

## 7. 反模式（出现即不合格）

- 没有自动化测试，或只有手测。
- 倒金字塔：大量 e2e、几乎无单测。
- 测私有实现细节，一重构全红。
- 单测里打真网络/真数据库/真第三方。
- 只测 happy path，不测错误与边界。
- flaky 测试长期"重跑就过"；测试相互依赖、有顺序。
- 为凑覆盖率写无断言/弱断言的"假测试"。

## 8. 最低交付 checklist

- [ ] 金字塔结构：单元为主、集成适量、e2e 少量覆盖关键流。
- [ ] 领域逻辑有单测；服务层用例 mock 依赖后测编排与错误；repository 用真 DB 集成测。
- [ ] API 层测状态码/校验/错误信封/鉴权。
- [ ] AAA 结构、一测一行为、覆盖正常+边界+错误。
- [ ] 测试独立、可并行、无 flaky；外部依赖用替身或一次性容器。
- [ ] CI 自动跑且失败阻断合并；关键路径覆盖达阈值。
