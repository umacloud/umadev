# 风险驱动测试矩阵（商业级必读）

> 测试的产能永远是有限的，"全都测、都测到 100%"既做不到也不划算。真正的问题不是"测得够不够多"，而是"有限的测试投入有没有压在最可能出事、出事最严重的地方"。对所有模块一刀切同等测试深度，结果往往是低风险代码被过度测试、高风险链路反而漏测。本矩阵把测试投入与**业务风险**对齐：先给每个变更/模块评风险，再按风险等级映射到差异化的测试深度与发布门禁，让回归更快、事故更少。

## 1. 风险评分模型（四因子）

对每个待测模块/变更，按四个因子各评 低(1)/中(2)/高(3)：

| 因子 | 含义 | 低(1) | 高(3) |
|---|---|---|---|
| 业务影响 | 出错对用户/收入/合规的伤害 | 内部辅助功能 | 支付/账号/数据/核心任务流 |
| 变更复杂度 | 改动的范围与不确定性 | 文案/样式微调 | 跨服务、并发、状态机、迁移 |
| 历史缺陷密度 | 该区域过去的故障频率 | 长期稳定 | 反复出问题的"重灾区" |
| 外部依赖数量 | 依赖的下游系统/三方数量 | 自包含 | 多三方、强网络、强时序耦合 |

**风险分 = 四因子之和（4–12）**，映射到风险等级：

| 风险分 | 等级 | 含义 |
|---|---|---|
| 4–6 | R1 低 | 出错影响小且不易错 |
| 7–9 | R2 中 | 需要常规多层验证 |
| 10–12 | R3 高 | 关键链路，需专项 + 门禁 |

> 任一因子=高 且 业务影响=高，直接按 R3 处理（一票升级），不必等总分达标——支付链路即使改动很小也是高风险。

## 2. 风险等级 → 测试深度映射

| 等级 | 必做测试 | 覆盖率要求 | 自动化 |
|---|---|---|---|
| R1 低 | 单元 + 冒烟 | 关键路径覆盖即可 | 可手动冒烟 |
| R2 中 | 单元 + 集成 + 契约 | 行/分支覆盖达团队基线 | 集成纳入 CI |
| R3 高 | 单元 + 集成 + 契约 + E2E + 性能 + 安全专项 + 故障注入 | 关键分支接近全覆盖，含异常/边界/并发路径 | 全自动 + 发布门禁 |

- **契约测试**对有外部依赖的模块是 R2 起步的硬要求，防止"各自单测都过、联调才炸"。
- **性能/安全专项**只对 R3 强制，避免在低风险模块上空耗。
- R3 必须显式覆盖**异常路径、边界值、并发与重试、幂等**，而不只是 happy path。

## 3. 发布门禁（风险等级决定阻断强度）

| 等级 | 回归失败处置 | 发布要求 |
|---|---|---|
| R1 | 修复或记录待办 | 冒烟通过即可放行 |
| R2 | 阻断本次发布，修复后重跑 | 集成+契约全绿 |
| R3 | 阻断 + 触发复盘 | 全部专项通过、关键缺陷清零、有回滚预案 |

- **高风险模块回归失败 = 禁止发布**，不接受"已知问题先上、下个版本修"。
- **高风险缺陷必须沉淀为长期回归用例**（缺陷驱动测试），防止同一根因反复复发。
- 风险评分是**动态的**：每次发布前根据新变更、新依赖、新故障重新评分，矩阵不更新就会失去参考价值。

## 4. 与变更频率结合（热点优先）

把"风险高"与"改得勤"叠加，识别需要最强测试纪律的热点：

- 高风险 × 高频变更 → 投入最深测试 + 强门禁 + 监控告警兜底。
- 高风险 × 低频变更 → 每次改动都需完整 R3 流程（改得少更易疏忽）。
- 低风险 × 高频变更 → 重自动化冒烟与回归，控制人工成本。

## 反模式

- **一刀切测试深度**：所有模块同等对待，低风险代码被过度测试、高风险链路反而漏测。
- **评分一次定终身**：风险分上线后从不更新，新增的依赖与故障未反映，矩阵沦为摆设。
- **只看总分忽略一票升级**：支付/账号类小改动因总分不高被降级处理，绕过专项。
- **覆盖率刷数字**：为达 R3 覆盖率指标补一堆只测 happy path 的用例，异常/并发/边界仍是盲区。
- **高风险失败放行**：以"deadline 到了"为由对 R3 回归失败放行，把风险转嫁给线上。
- **缺陷不沉淀**：修了线上缺陷却不补回归用例，同一根因下个版本再炸。

## 最低交付 checklist

- [ ] 每个待测模块/变更有四因子风险评分与 R1/R2/R3 等级，且本次发布前已刷新。
- [ ] 高业务影响项已按一票升级规则处理为 R3。
- [ ] 各等级的必做测试类型已按映射表执行（R2 起含契约、R3 含性能+安全+故障注入）。
- [ ] R3 用例显式覆盖异常路径、边界、并发/重试与幂等。
- [ ] 发布门禁按等级生效：R2/R3 回归失败阻断发布。
- [ ] 所有已修复的线上缺陷均已沉淀为长期回归用例。
- [ ] 高风险 × 高频变更的热点模块有自动化回归 + 监控告警兜底。
