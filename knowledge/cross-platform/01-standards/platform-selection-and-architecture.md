# 多端平台选型与架构标准（商业级必读）

> 商业项目不只有 web。先确定**目标平台**，再选技术栈与架构。架构文档必须明确：要做哪些端、用什么方案、哪些代码共享。本标准给出选型决策 + 跨端共享架构。

## 1. 先确定目标平台（架构文档必须声明）

| 平台 | 典型场景 |
|---|---|
| **Web**（浏览器）| 后台/SaaS/官网/工具 |
| **移动 App**（iOS/Android/鸿蒙）| 面向 C 端、需推送/相机/离线/原生体验 |
| **小程序**（微信/支付宝/抖音等）| 国内轻量、即用即走、依附超级 App 流量 |
| **桌面 App**（Win/macOS/Linux）| 生产力工具、需本地能力/离线/系统集成 |
| **多端统一** | 同时要 App+小程序+H5：用 uni-app/Taro/Flutter 等 |

## 2. 技术栈选型决策

- **移动端**：
  - 跨平台优先 **Flutter**（高性能自绘、一致体验）或 **React Native**（贴近 web 生态）。
  - 重原生体验/平台特性 → **原生**：iOS Swift/SwiftUI、Android Kotlin/Jetpack Compose。
  - 国内必须支持鸿蒙 → **ArkTS/ArkUI**（HarmonyOS NEXT 不兼容安卓，需单独适配）。
- **小程序**：单端用原生（微信原生）；多小程序+App+H5 用 **uni-app / Taro** 一套多端。
- **桌面端**：Web 技术栈复用 → **Tauri**（Rust，轻、安全、包小，推荐）或 **Electron**（生态成熟、包大）；重性能/系统集成 → 原生(SwiftUI/WinUI/Qt)。
- 不要无脑全平台原生分开写（成本×N）；也不要用跨平台硬上不适合的场景（如重图形/AR）。

## 3. 跨端共享架构（关键：分什么、不分什么）

**核心原则：业务逻辑/数据/契约共享，UI/交互按平台。**

```
┌─────────── 各平台 UI 层（按各端 HIG / 设计规范，不共享）───────────┐
│  Web React   │ iOS SwiftUI │ Android Compose │ 小程序 │ 鸿蒙 ArkUI │
└──────────────┴─────────────┴──────────────────┴────────┴───────────┘
            ▼ 调用 ▼
┌──── 共享层（尽量共享）：领域逻辑 · 用例/服务 · 数据模型(DTO) · API 契约 · 校验规则 ────┐
└──────────────────────────────────────────────────────────────────────────┘
            ▼
        统一后端 / BFF（按端裁剪聚合）
```

- **共享**：领域模型、业务规则、API 契约/客户端、校验逻辑、状态管理核心、常量。原生多端可用 KMP(Kotlin Multiplatform)/共享 SDK；JS 系可抽 npm 包。
- **不共享**：UI 组件、导航、动画、平台原生能力调用——这些**遵循各平台自己的设计与交互规范**（iOS HIG、Android Material、鸿蒙 HarmonyOS Design、小程序设计指南、桌面平台规范），别把一端的 UI 范式硬套到另一端。
- **后端**：同一套 API；多端差异大时用 **BFF（Backend for Frontend）**按端聚合裁剪。

## 4. 跨端通用 vs 平台特定

- **通用规范仍然适用**（无论哪端）：分层/服务层/数据建模/API/认证授权/安全/性能/测试/可观测——见 backend/frontend 各标准。客户端也要分层（UI/状态/数据访问/领域），数据访问统一走 API 层。
- **平台特定规范**（各端标准）：UI/交互范式、生命周期、导航、原生能力、打包发布、商店/平台审核、性能模型。

## 5. 反模式（出现即不合格）

- 架构文档不声明目标平台，默认只做 web。
- 把 web 的 UI/交互范式硬套到 App/小程序/桌面。
- 该共享的业务逻辑各端重复写；该按平台的 UI 强行共享导致体验割裂。
- 国内要求鸿蒙却只做安卓 APK（鸿蒙 NEXT 不兼容 apk）。
- 用跨平台框架硬做不适合的重原生/重图形场景。

## 6. 最低交付 checklist

- [ ] 架构文档明确目标平台 + 技术栈选型 + 理由。
- [ ] 业务逻辑/数据/契约/校验共享；UI/交互按各端设计规范。
- [ ] 客户端也分层（UI/状态/数据访问/领域），数据访问统一 API 层。
- [ ] 通用规范(分层/安全/性能/测试)跨端适用 + 各端平台特定规范遵守。
- [ ] 多端差异大时用 BFF；国内合规端（鸿蒙/小程序）单独适配。
