# 生产就绪审查规范（PRR · 商业级必读）

> "代码写完了"不等于"可以上线了"。一个服务能不能交付生产，取决于它能不能被**运维**：出问题看得见吗、扛得住吗、回得来吗、有人值班吗、有手册吗。
> 生产就绪审查（PRR）是上线前的一次**结构化 go/no-go 决策**——把 SLO、可观测性、回滚、安全、容量、值班、手册这些散落的线索汇到一张桌子上，做一次明确的"放行 / 不放行"判定，并留下可审计的签字。它不是又一份逐条勾选的清单（清单见 `development/03-checklists/production-readiness-checklist`），而是把那些维度**收口成一个决策**的流程与门。

## 1. 什么时候触发 PRR

| 触发场景 | 是否需要 PRR |
|---|---|
| 新服务首次上线 | 必须，完整 PRR |
| 重大架构变更 / 新增关键依赖 | 必须，聚焦受影响维度 |
| 承接显著增长流量 / 新区域部署 | 必须，重点容量与容灾 |
| 日常功能迭代（已就绪服务内） | 不需要，走常规发布门 |

PRR 审查的是**服务的可运维性**，不是这一次代码改动的正确性。后者由代码评审、测试、发布门禁覆盖；PRR 回答的是"这个服务上了生产，运营得起来吗"。

## 2. 心态：上线是一个决策，不是一个事件

- **可运维性优先于功能完整**：宁可少上一个功能，也不要上一个出了事看不见、回不来的服务。
- **未就绪要敢拦**：PRR 的价值在于敢说 no-go。橡皮图章式的 PRR 等于没有。
- **就绪是带条件的**：可以"有条件放行"——明确遗留项、责任人、期限与补救方案，而不是非黑即白。
- **审查留痕**：决策、未决项与签字归档，出事可回溯"当初是怎么放行的"。

## 3. 审查维度（七根支柱，逐一收口）

每一维度判定 **就绪 / 有条件 / 未就绪**，并指向其权威标准。

### 3.1 SLO 与服务目标
- 是否定义了**面向用户**的 SLI 与现实可达的 SLO（可用性/延迟/正确性）？错误预算如何使用？（见 `observability/01-standards/observability-and-slo-operations`、`operations/slo-sli-playbook`）
- 关键业务指标是否明确（不止技术指标）？

### 3.2 可观测性
- 日志结构化、指标（RED/USE + 业务）、追踪是否齐备且**可关联**（一个 request_id 串起一次请求）？
- 凭现有遥测能否定位**未预想**的故障，而不必先发版加日志？（见 `observability/01-standards/observability-and-slo-operations`）

### 3.3 告警与值班
- 告警是否**基于用户症状/错误预算燃尽**、条条可执行、指向 runbook、已分级？
- 是否有明确的**值班轮值 + 升级路径 + 关键联系人**，且通知链路端到端验证过？

### 3.4 回滚与恢复
- 是否有**已验证可执行**的回滚方案（应用/配置/数据），回滚和发布同等可靠？（见 `release-engineering/release-rollback-and-recovery-playbook`）
- 备份是否做了且**恢复演练过**（不是纸面备份）？关键数据 RPO/RTO 是否满足业务？（容灾见 `architecture/resilience-and-disaster-patterns`）

### 3.5 韧性与容量
- 出站调用是否有超时/重试预算/熔断/舱壁/降级？过载有无背压与卸载？（见 `backend/01-standards/resilience-and-fault-tolerance`）
- 是否做过容量评估与压测（≥ 预期峰值的余量）？关键依赖故障的影响面是否已知且可控？

### 3.6 安全与合规
- 认证/授权/密钥管理/输入校验/安全头是否到位？依赖漏洞是否已分诊处理？（见 `backend/01-standards/dependency-and-supply-chain-hygiene`）
- 审计日志、数据合规（个人数据处理）是否满足要求？

### 3.7 运维手册与所有权
- 是否有**Runbook**（常见故障的诊断与处置步骤，新人可照做）？
- 服务**owner 明确**、依赖清单与上下游 SLA 清楚、变更/回滚方式已文档化？

## 4. 决策判定与放行口径

| 整体判定 | 条件 | 动作 |
|---|---|---|
| Go（放行） | 所有支柱"就绪"，无 Blocker | 批准上线，归档签字 |
| Conditional Go（有条件放行） | 无 Blocker，存在带责任人+期限的遗留项 | 放行 + 跟踪遗留项到关闭 |
| No-Go（不放行） | 存在 Blocker（无回滚/无监控/无值班/未达 SLO 前提） | 拦截，修复后重审 |

**硬 Blocker（任一存在即 No-Go）**：没有可执行回滚；故障基本看不见（无遥测/无告警）；无人值班或升级路径不通；备份从未验证恢复；已知严重且可利用的安全漏洞未处理。这些是"上了就是赌运气"的红线。

## 5. 角色与产出

- **服务 owner**：发起 PRR，准备证据（SLO 定义、监控看板、回滚演练记录、runbook）。
- **审查者**：运维/SRE 视角 + 安全视角，逐维度判定，敢于给 No-Go。
- **产出物**：一份 PRR 记录——逐维度判定、遗留项（责任人/期限/补救）、最终 go/no-go 决策与签字，归档可审计（建议落 `output/`）。
- **闭环**：有条件放行的遗留项必须被跟踪到关闭；上线后一段时间复盘 SLO 实际表现与首批事故。

## 6. 反模式（出现即不合格）

1. **橡皮图章 PRR**：走过场签字，从不给 No-Go，等于没审查。
2. **代码评审冒充 PRR**：只审这次改动对不对，不审服务能不能被运维。
3. **没有回滚就上**：出事只能向前修，没有"撤回"按钮。
4. **遥测/告警缺位还放行**：上线后故障看不见，全靠用户投诉。
5. **无人值班 / 升级路径不通**：半夜出事没人响应，告警发进黑洞。
6. **备份没验证恢复**：以为有备份，真要恢复时发现恢复不出来。
7. **有条件放行后不跟踪**：遗留项一放就忘，技术债与风险沉淀。
8. **PRR 只在首次上线做一次**：重大架构变更后从不重审，老结论早已失效。

## 7. 最低交付 checklist

- [ ] 明确 PRR 触发条件，新服务/重大变更/扩量/新区域强制 PRR。
- [ ] 七根支柱逐一收口判定（就绪/有条件/未就绪），各指向权威标准并附证据。
- [ ] SLO 面向用户且现实；遥测可关联、能定位未预想故障；告警可执行分级。
- [ ] 回滚方案已验证可执行；备份已演练恢复；RPO/RTO 满足业务。
- [ ] 韧性（超时/熔断/降级）与容量（压测余量）已验证；依赖故障影响面可控。
- [ ] 值班轮值 + 升级路径 + 通知链路端到端验证；服务 owner 明确；Runbook 可照做。
- [ ] 给出明确 go / conditional-go / no-go 决策；硬 Blocker 一律 No-Go。
- [ ] PRR 记录（判定/遗留项/责任人/期限/签字）归档可审计；有条件放行项跟踪到关闭。
