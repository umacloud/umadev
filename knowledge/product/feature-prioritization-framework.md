# 功能优先级决策框架（商业级必读）

> 优先级决定一个团队把有限的研发产能投到哪里——它比"做得好不好"更早、更致命地决定产品成败。最常见的失败不是把某个功能做砸，而是把整季产能投在低价值需求上。"拍脑袋排期""谁声音大做谁的""老板说的先做"都不是优先级方法，只是把决策责任藏起来。本框架把优先级从主观争论升级为**可复算、可追责、带门禁的决策**：每个需求都要有量化打分、验收标准、回滚策略，且高风险项必须双签。

## 1. 先分类，再打分（不要对所有需求用同一把尺）

打分前先用一个粗筛把需求分桶，避免把"救火"和"探索"放进同一个排序里：

| 类型 | 判定 | 处置 |
|---|---|---|
| **合规/安全/止血** | 不做会违法、泄露、资损或线上事故 | 不进打分队列，直接最高优先级排期 |
| **核心价值线** | 直接驱动北极星指标（激活/留存/收入/成本） | 进加权打分，主战场 |
| **体验/效率优化** | 改善现有任务流的成功率与耗时 | 进打分，与价值线竞争产能 |
| **探索/赌注** | 高不确定性、可能高回报 | 单独留固定产能比例（如 10–20%），不与确定性需求直接比分 |
| **债务/可维护性** | 不做会持续拖慢未来交付 | 量化"拖慢成本"后进打分，禁止长期归零 |

## 2. 四维加权评分模型

每个进入打分队列的需求按四维各 1–5 分，给出可复算的综合分。

| 维度 | 含义 | 1 分 | 5 分 | 默认权重 |
|---|---|---|---|---|
| 业务价值 | 对收入/留存/成本/战略的直接推动 | 几乎无关 | 直接撬动北极星指标 | 0.35 |
| 用户价值 | 解决痛点的频率 × 强度 | 边缘场景 | 高频关键任务阻塞 | 0.25 |
| 实现成本 | 研发+测试+运维+迁移+培训（反向计入） | 一周内 | 跨团队多季度 | 0.20 |
| 风险敞口 | 合规/安全/稳定性/依赖不确定性（反向计入） | 隔离可控 | 触及账号/支付/数据 | 0.20 |

`综合分 = (业务价值×0.35 + 用户价值×0.25) − (实现成本×0.20 + 风险敞口×0.20)`

- 权重不是固定的：早期产品调高用户价值，规模化阶段调高业务价值与风险；权重一旦定了，**整季不随单个需求改**，否则打分失去公信力。
- 分数相近（差值在 0.3 内）时，**以"减少关键链路风险/降低不确定性"优先**——先做能解锁后续决策的需求。
- 打分是输入不是判决：评审用分数聚焦讨论分歧最大的维度，而不是机械按分排期。

## 3. 价值/成本四象限（快速版，用于粗排与对齐）

当需求多、时间紧时，用二维矩阵先对齐方向，再对争议项做四维细打分。

| | 低成本 | 高成本 |
|---|---|---|
| **高价值** | 立即做（quick win） | 重点投入，需拆分里程碑 |
| **低价值** | 顺手做/批量清 | 不做或砍掉（最危险象限：高成本低价值） |

- 警惕"高成本低价值"象限——它最容易因为"已经投入了"而被沉没成本绑架。
- "高价值高成本"必须拆成可独立交付、可独立验证的里程碑，避免一次性大爆炸。

## 4. 满意度模型：区分"必备"与"惊喜"

同样高分的需求，对用户满意度的作用并不线性，排期时要区分：

- **必备型**：缺了就投诉、做到顶也只是"不扣分"（如登录稳定、支付不丢单）→ 必须满足基线，但不要过度打磨。
- **期望型**：投入与满意度线性相关 → 主战场，按四维分排。
- **惊喜型**：用户没预期，做到了显著加分 → 适合作为差异化赌注，但不能挤占必备型产能。

## 5. 决策门禁（硬约束，不可绕过）

- 涉及**账号、支付、权限、数据导出/删除**的功能，风险评分必须由**安全 + 运维双签**方可进开发。
- 涉及**跨系统/跨团队**变更，必须附**回滚方案 + 向后兼容策略**，否则不予排期。
- **没有可量化验收标准**的需求不进开发队列（"做个更好的搜索"不是需求，"搜索 P95 < 300ms 且 0 结果率 < 5%"才是）。
- 每个排期需求必须绑定**至少 1 个业务指标 + 1 个质量指标**，并预设上线后的验证窗口与下线条件。

## 6. 输出模板（每个排期需求必须填全）

- 需求名称与一句话价值主张
- 预期业务影响（绑定哪个指标、预期幅度、衡量口径）
- 受益用户群体与触发场景频率
- 四维打分 + 综合分（含权重快照）
- 估算工作量与关键依赖
- 风险敞口与缓解措施（高风险需双签记录）
- 验收标准（可判定）+ 上线后验证窗口 + 回滚/下线条件
- 发布策略（灰度比例、放量节奏）

## 反模式

- **拍脑袋优先级**：用职级、嗓门、最近一次会议决定排期，没有可复算的打分，事后无法追责。
- **权重随需求漂移**：为了让某个需求排上去临时改权重，打分体系沦为表演。
- **只算价值不算风险**：把账号/支付/权限类高风险需求当普通功能排，绕过双签，埋下资损与合规雷。
- **沉没成本绑架**：因为"已经做了一半"就继续投入高成本低价值项，而不是及时止损。
- **无验收标准入队**：需求描述模糊、没有可量化的成功定义，导致做完无法判断成败，也无法回滚。
- **探索与确定性混排**：把高不确定性赌注和确定性需求放进同一个排序，要么赌注永远排不上、要么挤垮基本盘。
- **必备型过度打磨**：在"做到顶也只是不扣分"的必备功能上投入惊喜型的产能。

## 最低交付 checklist

- [ ] 所有候选需求已分桶（合规止血 / 核心价值 / 体验优化 / 探索 / 债务），止血类不进打分直接排期。
- [ ] 进队列需求均有四维打分 + 综合分，权重在本季度内固定且公开。
- [ ] 高成本需求已拆为可独立交付、可独立验证的里程碑。
- [ ] 账号/支付/权限/数据类需求已完成安全+运维双签并留痕。
- [ ] 跨系统变更均附回滚方案与向后兼容策略。
- [ ] 每个排期需求绑定 ≥1 业务指标 + ≥1 质量指标，并有可判定的验收标准。
- [ ] 探索类需求有独立产能配额，未挤占确定性基本盘。
- [ ] 排期结论与未采纳需求的理由均有记录，可向相关方解释。
