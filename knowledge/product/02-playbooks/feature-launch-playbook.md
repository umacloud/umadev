# 功能上线 Playbook

## 概述

本 Playbook 定义功能从开发完成到全量上线的标准流程。适用于面向用户的产品功能发布，
覆盖灰度策略、监控配置、数据验证、全量发布和上线复盘五个阶段。
目标：确保每次功能上线可控、可观测、可回滚。

---

## 阶段一：上线准备（发布前 3-5 天）

### 1.1 功能验收

- [ ] 产品经理完成功能验收并签字确认
- [ ] QA 完成全量测试用例执行，通过率 100%
- [ ] 无 P0/P1 级别的已知 Bug
- [ ] 性能测试通过（目标接口 P99 < 500ms，QPS 满足预估峰值 1.5x）
- [ ] 安全审查通过（无 SQL 注入/XSS/越权等高危漏洞）

### 1.2 发布计划

- [ ] 发布日期和时间窗口确定（避开业务高峰期）
- [ ] 灰度策略确定（用户比例/用户分组/地区）
- [ ] 回滚方案文档化（触发条件 + 操作步骤 + 预期耗时）
- [ ] 相关团队通知（客服/运营/市场）并同步 FAQ

### 1.3 监控准备

- [ ] 功能相关的业务指标埋点已上线并验证
- [ ] 监控 Dashboard 已创建（核心指标：转化率/错误率/性能）
- [ ] 告警规则已配置（阈值基于历史基线）
- [ ] 日志关键字监控已配置（ERROR/WARN 相关的新增日志）

### 1.4 Feature Flag 配置

- [ ] Feature Flag 已创建并在 staging 环境验证
- [ ] Flag 默认关闭，灰度规则已配置
- [ ] 紧急关闭 Flag 的操作步骤已验证（< 1 分钟可完成）
- [ ] Flag 清理计划已排期（全量后 2 周内清除）

---

## 阶段二：灰度发布（第 1-3 天）

### 2.1 灰度策略推荐

```
阶段 1: 内部员工 / 种子用户（1-2%）  →  观察 24h
阶段 2: 5% 用户                      →  观察 24-48h
阶段 3: 20% 用户                     →  观察 24h
阶段 4: 50% 用户                     →  观察 12-24h
阶段 5: 100% 全量
```

### 2.2 灰度期间检查项

每个灰度阶段推进前，需确认以下指标正常：

| 检查项 | 达标标准 | 检查频率 |
|--------|----------|----------|
| 接口错误率 | 与基线持平（波动 < 0.5%） | 每小时 |
| 接口 P99 延迟 | 与基线持平（波动 < 20%） | 每小时 |
| 核心转化率 | 不低于基线 5% | 每 4 小时 |
| 客诉量 | 无新增相关客诉 | 每 4 小时 |
| 资源使用 | CPU/内存无异常飙升 | 每小时 |

### 2.3 灰度期间沟通

- 每日发布灰度状态报告到项目群
- 报告包含：当前灰度比例、核心指标、异常情况、下一步计划
- 异常情况 30 分钟内通知项目负责人

---

## 阶段三：数据验证（灰度期间持续进行）

### 3.1 功能指标验证

- [ ] 功能使用率是否达到预期（DAU 渗透率）
- [ ] 关键路径转化漏斗是否符合预期
- [ ] 新功能是否对现有功能产生负面影响（回归指标）
- [ ] A/B 实验数据是否具有统计显著性（p < 0.05）

### 3.2 技术指标验证

- [ ] 新增接口性能是否达标
- [ ] 数据库查询是否有慢查询（> 1s）
- [ ] 缓存命中率是否正常
- [ ] 异步任务（队列/定时任务）是否正常消费

### 3.3 数据一致性

- [ ] 新功能产生的数据格式和内容正确
- [ ] 数据写入路径（DB/ES/缓存）一致性检查
- [ ] 报表和统计数据是否准确反映新功能数据
- [ ] 与上下游系统的数据同步是否正常

---

## 阶段四：全量发布

### 4.1 全量发布决策

**决策者**: 产品负责人 + 技术负责人 联合确认

**必须满足的前提条件**:
- [ ] 灰度期间所有检查项达标
- [ ] 无未解决的 P0/P1 问题
- [ ] 数据验证报告确认功能指标达预期
- [ ] 客服团队已完成培训并准备好 FAQ

### 4.2 全量操作

- [ ] 确认发布时间窗口（建议工作日上午 10:00-11:00）
- [ ] Feature Flag 切换到 100%
- [ ] 全量后 2 小时内密切监控核心指标
- [ ] 确认无异常后通知相关团队"全量发布完成"

### 4.3 全量后观察（48 小时）

- [ ] 持续监控核心业务指标（日报频率）
- [ ] 关注客诉渠道是否有新增反馈
- [ ] 观察系统资源使用趋势（是否有缓慢增长）
- [ ] 确认数据报表和统计正确

---

## 阶段五：上线复盘（全量后 1 周内）

### 5.1 复盘会议议程

```
1. 功能概述和目标回顾             (5 min)
2. 上线过程回顾（时间线/灰度/问题）  (15 min)
3. 数据结果 vs 预期目标             (10 min)
4. 问题和经验总结                   (15 min)
5. 后续优化计划                     (10 min)
```

### 5.2 复盘文档模板

```markdown
## 功能名称: ___
## 上线日期: ___
## 负责人: ___

### 目标达成情况
| 指标 | 目标值 | 实际值 | 达标 |
|------|--------|--------|------|

### 上线过程中的问题
| 问题 | 影响 | 处理方式 | 改进建议 |
|------|------|----------|----------|

### 经验总结（做得好的 / 需要改进的）
### 后续优化 Backlog
```

### 5.3 清理工作

- [ ] Feature Flag 代码清理（全量稳定 2 周后）
- [ ] 临时监控告警调整为长期告警
- [ ] 文档更新（用户文档/API 文档/内部 Wiki）
- [ ] 复盘文档归档

---

## 紧急回滚流程

### 回滚触发条件（满足任一即触发）

- 核心转化率下降 > 10%
- 接口错误率上升 > 2%
- P0 级 Bug 影响核心功能
- 大量用户投诉（30 分钟内 > 10 条）

### 回滚操作

```
1. Feature Flag 紧急关闭（< 1 分钟）
2. 确认功能已下线（检查监控）
3. 通知项目组 + 客服团队
4. 排查根因并修复
5. 修复后重新走灰度流程
```

## Agent Checklist

- [ ] 发布前是否完成功能验收和安全审查
- [ ] 灰度策略是否分阶段推进（而非一次全量）
- [ ] 监控 Dashboard 和告警是否在发布前就绪
- [ ] Feature Flag 紧急关闭是否经过验证
- [ ] 回滚触发条件和操作步骤是否明确
- [ ] 全量发布是否有明确的决策人和决策标准
- [ ] 上线复盘是否在 1 周内完成
- [ ] Flag 代码清理是否已排入 Backlog
