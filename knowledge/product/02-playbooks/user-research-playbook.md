# 用户研究作战手册 (User Research Playbook)

## 概述

本手册定义了用户研究的完整流程，涵盖研究计划制定、用户访谈、可用性测试、问卷调查、数据分析和洞察报告。目标是通过系统化的研究方法获取真实用户洞察，驱动产品决策，减少拍脑袋式的设计和开发。

## 前置条件

### 必须满足

- [ ] 已明确研究目标和核心问题（要验证什么假设 / 要回答什么问题）
- [ ] 已确定研究预算和时间范围
- [ ] 已获得团队对研究计划的认可
- [ ] 有至少一位经验丰富的研究者主导

### 建议满足

- [ ] 有用户画像（Persona）作为招募基础
- [ ] 有竞品分析报告作为参考
- [ ] 有历史用户数据（行为数据、客服工单、NPS 反馈）
- [ ] 已建立用户研究知情同意和隐私保护流程

## 步骤一：研究计划

### 1.1 研究计划模板

```markdown
# 用户研究计划：[项目名称]

## 1. 背景与目标
- 业务背景：[简述产品/功能背景和当前问题]
- 研究目标：[明确 2-3 个核心目标]
- 核心问题：
  1. [需要回答的问题 1]
  2. [需要回答的问题 2]
  3. [需要回答的问题 3]

## 2. 研究假设
- H1: [假设 1 -- 预期用户行为/态度]
- H2: [假设 2 -- 预期用户行为/态度]
- H3: [假设 3 -- 预期用户行为/态度]

## 3. 研究方法
| 方法 | 目的 | 参与人数 | 时长 |
|------|------|---------|------|
| 用户访谈 | 探索用户需求和痛点 | 8-12 人 | 各 45-60 分钟 |
| 可用性测试 | 验证设计方案的易用性 | 5-8 人 | 各 30-45 分钟 |
| 问卷调查 | 量化验证定性发现 | 200+ 人 | 约 10 分钟 |

## 4. 目标用户
- 用户画像 A：[描述]，招募 X 人
- 用户画像 B：[描述]，招募 Y 人

## 5. 时间线
- Week 1: 计划制定 + 招募
- Week 2: 执行访谈/测试
- Week 3: 数据分析
- Week 4: 报告产出 + 分享

## 6. 交付物
- 研究洞察报告 (PDF/Notion)
- 关键发现卡片 (Miro/FigJam)
- 设计建议清单
- 原始数据归档
```

### 1.2 研究方法选择矩阵

| 场景 | 推荐方法 | 样本量 | 产出 |
|------|---------|--------|------|
| 探索新方向，不确定用户需要什么 | 深度访谈 | 8-15 人 | 用户需求和痛点地图 |
| 验证设计方案是否好用 | 可用性测试 | 5-8 人 | 问题列表和改进建议 |
| 量化已知问题的严重程度 | 问卷调查 | 200+ 人 | 统计数据和优先级排序 |
| 理解用户真实使用行为 | 行为数据分析 | 全量用户 | 漏斗转化和使用模式 |
| 对比两个设计方案 | A/B 测试 | 1000+ 人 | 统计显著性结论 |
| 了解用户日常工作/生活 | 日记研究 | 10-15 人 | 使用情境和习惯洞察 |
| 快速验证概念方向 | 概念测试 | 8-12 人 | 概念吸引力和改进方向 |

## 步骤二：用户访谈

### 2.1 招募

#### 招募渠道

| 渠道 | 适用场景 | 时间 | 成本 |
|------|---------|------|------|
| 产品内弹窗邀请 | 活跃用户 | 3-5 天 | 低 |
| 客户成功团队推荐 | B2B 关键客户 | 5-7 天 | 低 |
| 社交媒体/社群 | 潜在用户 | 3-7 天 | 中 |
| 专业招募平台 (UserTesting / 问卷星) | 特定画像用户 | 1-3 天 | 高 |
| 线下场所拦截 | 大众消费场景 | 1-2 天 | 中 |

#### 招募筛选问卷

```markdown
# 筛选问卷（示例：电商产品）

1. 您的年龄段？
   - [ ] 18-24  - [ ] 25-34  - [ ] 35-44  - [ ] 45-54  - [ ] 55+

2. 您最近 3 个月在网上购物的频率？
   - [ ] 每周多次  - [ ] 每周一次  - [ ] 每月 2-3 次  - [ ] 每月一次以下

3. 您最常使用的购物平台？（多选）
   - [ ] 淘宝  - [ ] 京东  - [ ] 拼多多  - [ ] 抖音商城  - [ ] 其他：___

4. 您是否有以下经历？（筛选关键行为）
   - [ ] 在购物车中放置商品后最终没买
   - [ ] 因为找不到想要的商品而放弃购物
   - [ ] 因为退换货流程复杂而不满

5. 您是否愿意参加约 45 分钟的线上访谈？（补贴 ¥XXX）
   - [ ] 是  - [ ] 否
```

#### 招募注意事项
- 招募人数 = 实际需求 x 1.5（考虑爽约率 30%）
- 确认用户的可用时段和设备（线上需有稳定网络 + 摄像头）
- 签署知情同意书（包括录音录像授权、数据用途说明）
- 避免招募到"专业受试者"（经常参加各种调研的人）

### 2.2 访谈问题设计

#### 问题设计原则

| 原则 | 正确 | 错误 |
|------|------|------|
| 开放式提问 | "请描述一下您上次网购的经历" | "您网购体验好不好？" |
| 关注行为而非观点 | "您当时做了什么？" | "您觉得这个功能有用吗？" |
| 避免引导 | "您如何看待这个流程？" | "您不觉得这个流程太复杂了吗？" |
| 追问细节 | "能详细说说为什么吗？" | 接受表面回答就跳到下一题 |
| 避免假设性问题 | "您上次退货是怎么操作的？" | "如果有一键退货您会用吗？" |

#### 访谈大纲模板（45 分钟）

```markdown
# 访谈大纲：[主题]

## 暖场 (5 min)
- 自我介绍，说明访谈目的和录音事宜
- "先请简单介绍一下您自己，您的职业和日常工作？"
- "您使用 [产品类别] 大约多长时间了？"

## 背景了解 (10 min)
- "请描述一下您典型的 [使用场景]，通常在什么情况下会用到？"
- "您目前使用什么工具/产品来完成 [任务]？"
- "在这个过程中，什么让您最困扰？"
  - 追问："能举个具体的例子吗？"
  - 追问："这种情况多久出现一次？"
  - 追问："您当时是怎么解决的？"

## 核心探索 (20 min)
- "请回忆一下您最近一次 [核心行为] 的完整过程"
  - 追问起始："是什么触发了这个行为？"
  - 追问过程："中间有没有遇到什么困难？"
  - 追问结果："最终结果满意吗？"
  - 追问情感："当时的感受是什么？"

- [针对具体假设的探索问题]
  - "在 [场景] 中，您通常会怎么做？"
  - "您觉得 [现有方案] 最大的问题是什么？"
  - "如果要改善 [问题]，您觉得什么最重要？"

## 竞品参考 (5 min)
- "您有没有用过其他类似的产品？"
- "跟 [竞品] 相比，您觉得有什么不同？"
- "其他产品有什么让您印象深刻的？"

## 收尾 (5 min)
- "关于 [主题]，还有什么是您想补充的？"
- "如果只能改变一件事来改善您的体验，您希望是什么？"
- 感谢 + 补贴发放说明
```

### 2.3 访谈执行

#### 执行清单

```
访谈前
├── 测试录音/录像设备和网络
├── 准备访谈大纲和记录模板
├── 确认用户已签知情同意书
└── 安排记录员（或 AI 转录工具）

访谈中
├── 开场说明：目的、时长、录音、无对错之分
├── 从简单问题开始，建立信任
├── 保持中立表情和语气，不评价用户回答
├── 80/20 法则：用户说 80%，研究者说 20%
├── 记录关键引述（原话，带时间戳）
├── 观察非语言信号（迟疑、皱眉、叹气）
└── 预留最后 5 分钟做开放式收集

访谈后（30 分钟内）
├── 整理访谈笔记，补充观察备忘
├── 标记 3-5 个关键发现（印象最深的）
├── 归档录音/录像文件
└── 更新招募和执行进度表
```

#### 常见陷阱与应对

| 陷阱 | 表现 | 应对 |
|------|------|------|
| 用户给出期望答案 | "这个功能挺好的"（敷衍） | 追问具体使用场景和频率 |
| 用户漫谈偏题 | 花 10 分钟讲无关故事 | "非常有意思，我想回到 [主题]..." |
| 用户过于沉默 | 只回答"是/否" | 换成更具体的情境问题 |
| 用户提出解决方案 | "你们应该加个 XX 按钮" | "什么情况让您觉得需要这个功能？" |
| 研究者引导回答 | "所以您觉得这很麻烦对吧？" | 改为"您对这个过程有什么感受？" |

## 步骤三：可用性测试

### 3.1 测试准备

```markdown
# 可用性测试计划

## 测试对象
- 原型/产品版本：[Figma Prototype v2.1 / 线上版本]
- 测试设备：[桌面浏览器 / 移动端 / 两者]

## 测试任务（3-5 个核心任务）

### 任务 1：注册并完成首次设置
- 场景描述："您第一次使用这个产品，请完成注册并设置您的偏好"
- 成功标准：5 分钟内完成注册 + 至少设置 2 项偏好
- 观察重点：注册流程的跳出点、偏好设置的可理解性

### 任务 2：搜索并购买一件商品
- 场景描述："请搜索一件 200 元以内的蓝色 T 恤，并完成购买"
- 成功标准：3 分钟内找到目标商品并进入支付页
- 观察重点：搜索 + 筛选的使用路径、商品详情页的信息充分性

### 任务 3：处理异常场景
- 场景描述："您收到了一件尺码不合的衣服，请发起退换货"
- 成功标准：找到退换货入口并成功提交申请
- 观察重点：入口发现率、流程的清晰度

## 数据收集
- 任务完成率（Completion Rate）
- 任务完成时间（Time on Task）
- 错误次数（Error Count）
- 任务后满意度评分（1-7 单项满意度量表）
- 出声思维法（Think Aloud）记录
```

### 3.2 测试执行

#### 出声思维法（Think Aloud Protocol）

引导用户在操作过程中说出自己的想法：

```
引导话术：
"在接下来的操作中，请您把脑海中的想法说出来。
比如：'我在找退货按钮，我觉得应该在订单里面...'
'这个图标我不太确定是什么意思...'
'我以为点这个会跳到...'
这些都是非常有价值的反馈，没有对错之分。"
```

#### 观察记录模板

```markdown
| 时间 | 用户操作 | 用户说的 | 观察者备注 |
|------|---------|---------|-----------|
| 0:30 | 点击搜索框 | "我想搜蓝色T恤" | 直接输入，未使用分类导航 |
| 0:45 | 输入"蓝色T恤" | "结果好多啊" | 没注意到左侧筛选栏 |
| 1:10 | 向下滚动3屏 | "怎么按价格排序？" | 排序在页面顶部，用户已滚过 |
| 1:30 | 滚回顶部找排序 | (沉默，皱眉) | 排序下拉框不够明显 |
| 1:45 | 点击价格排序 | "找到了" | 花了1分钟才找到，期望30秒 |
```

### 3.3 可用性问题分级

| 级别 | 定义 | 影响 | 处理优先级 |
|------|------|------|-----------|
| 致命 (Critical) | 用户无法完成任务 | 用户流失 | 必须在上线前修复 |
| 严重 (Major) | 用户能完成但非常困难/需多次尝试 | 转化率下降 | 必须在当前版本修复 |
| 一般 (Minor) | 用户能完成但体验不佳/效率低 | 满意度下降 | 下个版本修复 |
| 建议 (Cosmetic) | 不影响使用但影响感受 | 品牌印象 | 排入 Backlog |

## 步骤四：问卷调查

### 4.1 问卷设计

#### 问题类型与适用场景

| 问题类型 | 示例 | 适用场景 |
|---------|------|---------|
| 李克特量表 (1-5/1-7) | "您对购物体验的满意度？" | 态度/满意度测量 |
| NPS (0-10) | "您有多大可能推荐给朋友？" | 忠诚度/口碑倾向 |
| 单选题 | "您最常使用的支付方式？" | 偏好/行为分布 |
| 多选题 | "以下哪些功能您使用过？" | 功能使用广度 |
| 矩阵题 | "对以下功能的满意度？" | 多维度批量评价 |
| 开放题 | "您最希望改进什么？" | 定性补充（放在最后） |
| SUS (System Usability Scale) | 10 题标准量表 | 系统可用性基准评分 |

#### 问卷设计原则

```
1. 总时长控制在 5-10 分钟（15-25 题）
2. 从简单客观题开始，逐步到复杂主观题
3. 避免双重否定：✗ "您不觉得这个功能不好用吗？"
4. 避免引导性：✗ "我们优秀的客服让您满意吗？"
5. 选项互斥且穷尽：提供"其他"选项
6. 敏感问题放后面，提供"不方便回答"选项
7. 关键问题设置反向题（attention check）
8. 必答题控制在 60% 以内，避免疲劳放弃
```

### 4.2 样本量参考

| 研究目的 | 最小样本量 | 建议样本量 | 备注 |
|---------|-----------|-----------|------|
| NPS 测量 | 100 | 200-400 | 需要按细分群体分析则更多 |
| 满意度调查 | 50 | 150-300 | 置信度 95%，误差 5% |
| A/B 偏好比较 | 每组 30 | 每组 100+ | 需要统计检验显著性 |
| 探索性调查 | 30 | 50-100 | 目的是发现模式，非精确测量 |

## 步骤五：数据分析

### 5.1 定性数据分析（访谈/可用性测试）

#### 亲和图法（Affinity Diagram）

```
Step 1: 提取原始数据点
  - 遍历所有访谈笔记和录音转写
  - 每个独立的观点/行为/引述 -> 一张便签
  - 保留用户原话（引号标注）
  - 标注来源（P1 = 第 1 位参与者）

Step 2: 自底向上分组
  - 将含义相近的便签归为一组
  - 为每组命名（主题标签）
  - 常见分组维度：用户目标/痛点/期望/行为模式/情感反应

Step 3: 提炼洞察
  - 每个主题组 -> 一个洞察陈述
  - 格式：[用户类型] 在 [场景] 中感到 [痛点/需求]，因为 [原因]
  - 标注支撑数据量（N=X 位用户提及）

Step 4: 优先级排序
  - 频率（多少用户提到）x 严重程度（对体验的影响）
  - 高频 + 高严重 = 必须解决
  - 高频 + 低严重 = 优化体验
  - 低频 + 高严重 = 监控关注
```

#### 洞察陈述示例

```markdown
### 洞察 1：新用户在注册后不知道下一步做什么 (N=7/10)

**发现**：70% 的新用户完成注册后在首页停留超过 30 秒，不知道第一步操作是什么。

**用户引述**：
- "注册完了然后呢？页面上东西太多了，不知道从哪开始" (P2)
- "我以为会有个教程之类的" (P5)
- "找了半天才发现要先创建项目" (P8)

**影响**：注册后 7 天留存率仅 15%，可能与首次体验困惑有关。

**建议**：添加新手引导（Onboarding Flow），包含 3-5 步核心操作引导。
```

### 5.2 定量数据分析（问卷/行为数据）

```python
# 常用分析方法

# 1. 描述性统计
import pandas as pd

df = pd.read_csv("survey_results.csv")
print(df["satisfaction"].describe())     # 均值/中位数/标准差
print(df["satisfaction"].value_counts()) # 频率分布

# 2. NPS 计算
promoters = len(df[df["nps_score"] >= 9])
detractors = len(df[df["nps_score"] <= 6])
total = len(df)
nps = (promoters - detractors) / total * 100
print(f"NPS: {nps:.0f}")

# 3. 交叉分析
pd.crosstab(df["user_type"], df["satisfaction"], normalize="index")
# 查看不同用户类型的满意度分布差异

# 4. 统计显著性检验
from scipy import stats
group_a = df[df["variant"] == "A"]["conversion"]
group_b = df[df["variant"] == "B"]["conversion"]
t_stat, p_value = stats.ttest_ind(group_a, group_b)
print(f"p-value: {p_value:.4f}")  # < 0.05 则差异显著

# 5. SUS 评分计算
# 奇数题：得分 - 1；偶数题：5 - 得分；总和 x 2.5
sus_scores = []
for _, row in df.iterrows():
    odd_sum = sum(row[f"sus_{i}"] - 1 for i in [1, 3, 5, 7, 9])
    even_sum = sum(5 - row[f"sus_{i}"] for i in [2, 4, 6, 8, 10])
    sus_scores.append((odd_sum + even_sum) * 2.5)
print(f"SUS Average: {sum(sus_scores)/len(sus_scores):.1f}")
# SUS > 68 = 可用性高于平均; > 80 = 优秀
```

## 步骤六：洞察报告

### 6.1 报告结构

```markdown
# 用户研究洞察报告：[项目名称]
日期：YYYY-MM-DD | 研究者：[姓名]

## 执行摘要（1 页）
- 研究背景和目标
- 方法概述（N 人访谈 + N 人可用性测试 + N 份问卷）
- 3-5 个核心发现（一句话 + 严重程度）
- 总体建议

## 研究方法
```
